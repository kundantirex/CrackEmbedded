C++ Basics

History & Program Structure

Developed by Bjarne Stroustrup in 1979 as "C with Classes". Adds OOP, templates, and exception handling to C.

Namespaces

Prevents name conflicts by grouping entities (classes, functions, variables) under a specific name (e.g., std::). If two libraries have a print() function, namespaces keep them separate.

cin / cout & References

cin reads input, cout prints output using << and >> operators.

References (&): An alias (another name) for an existing variable. Must be initialized when declared and cannot be changed to refer to something else. Safer than pointers (cannot be null).

Type Casting & const/constexpr

C++ style casts: static_cast (normal conversion), dynamic_cast (safe downcasting in inheritance), const_cast (removing constness), reinterpret_cast (dangerous, low-level bit reinterpretation).

const: Value cannot be modified. constexpr: Value must be evaluated at compile-time (improves performance).

OOP Concepts

  • Class & Object: A class is a blueprint. An object is an instance of a class.
  • Encapsulation: Bundling data and functions into a single unit (class) and hiding the internal state using access modifiers (private/protected).
  • Abstraction: Hiding complex implementation details and showing only the essential features to the user (achieved via interfaces/abstract classes).
  • Inheritance: A class derives properties from another. Types: Single, Multiple (two base classes), Multilevel (A->B->C), Hierarchical (one base, many derived), Hybrid.
  • Polymorphism: "Many forms". Compile-time: Function & Operator overloading. Run-time: Virtual functions (Overriding).
Association, Aggregation, Composition

Association: General relationship (A uses B).
Aggregation: Weak relationship (A "has-a" B). If A is destroyed, B can still exist (e.g., Department and Teachers).
Composition: Strong relationship (A "owns-a" B). If A is destroyed, B is also destroyed (e.g., House and Rooms).

Constructors & Destructors

A constructor initializes an object. It has the same name as the class and no return type.

  • Default: Takes no arguments.
  • Parameterized: Takes arguments to initialize members.
  • Copy Constructor: Initializes an object using another object of the same class MyClass(const MyClass& other).
  • Move Constructor: Transfers ownership of resources from a temporary (rvalue) object to a new object, avoiding deep copies MyClass(MyClass&& other).
  • Delegating: A constructor calling another constructor in the same class.
  • Destructor (~MyClass): Called automatically when the object goes out of scope. Used to free resources.
  • Initialization List: Used to initialize const members, reference members, and base classes before the constructor body runs.
  • Explicit: Prevents the compiler from using a single-argument constructor for implicit type conversions.
Rule of 3/5/0
Rule of 3: If you write a Destructor, Copy Constructor, or Copy Assignment, you probably need all three.
Rule of 5: C++11 added Move Constructor and Move Assignment to the Rule of 3.
Rule of 0: Ideally, design classes so you don't need any of them (use smart pointers and standard containers).

Operator Overloading

Gives special meaning to standard operators (like +, ==) for user-defined classes.

  • Rules: Cannot overload ., ::, sizeof, ?:. Cannot create new operators.
  • Unary/Binary: Unary operates on one operand (e.g., ++), Binary operates on two (e.g., +).
  • Friend Operators: Overloading << and >> (for cin/cout) must be done using friend functions because the left operand is ostream/istream, not your class.

Functions

  • Function Overloading: Functions with the same name but different parameters (number or type). Return type alone doesn't count.
  • Default Arguments: Must be assigned from right to left in the parameter list.
  • Inline Functions: Compiler expands the code at the call site. Good for small functions to save call overhead.
  • Templates: Write a function once that works for any data type template <typename T>.
  • Lambda Expressions: Anonymous inline functions. Syntax: [capture](params) { body }. Highly used with STL algorithms (e.g., std::sort).

Memory Management

  • new / delete: new allocates memory on the Heap and calls the constructor. delete calls the destructor and frees the memory.
  • new[] / delete[]: For arrays. Mismatching them (using delete on new[]) causes Undefined Behavior.
  • Placement new: Constructs an object at a pre-allocated memory address (useful in embedded systems).
  • Memory Leaks: Occurs when memory is allocated on the heap but not deleted.
  • RAII (Resource Acquisition Is Initialization): The most important idiom in C++. Tie the lifecycle of a resource (memory, file, mutex) to an object's scope. When the object goes out of scope, its destructor automatically frees the resource.

Smart Pointers (Very Important)

Wrappers around raw pointers that automatically manage memory via RAII. Included in <memory>.

PointerOwnershipCopy?Use Case
unique_ptrExclusiveNo (Move only)Single owner. Best performance (zero overhead).
shared_ptrSharedYesMultiple owners. Uses reference counting.
weak_ptrNon-owningYesUsed with shared_ptr to break circular dependencies.
  • make_unique / make_shared: Always prefer these over raw new. They are exception-safe and faster (especially make_shared which allocates the control block and object together).
  • Circular Dependency: If A has a shared_ptr to B, and B has a shared_ptr to A, the reference count never drops to 0. Memory leak! Fix: Make one a weak_ptr.
  • Custom Deleter: You can pass a custom function to delete the resource (e.g., closing a file instead of freeing memory).

Virtual Functions (Very Important)

Allows Runtime Polymorphism (dynamic binding). When a base class pointer points to a derived class object, calling a virtual function executes the derived class's version.

  • Pure Virtual Function: virtual void draw() = 0; Forces the derived class to implement it.
  • Abstract Class: A class with at least one pure virtual function. You cannot instantiate an abstract class.
  • vtable (Virtual Table): An array of function pointers created by the compiler for a class containing virtual functions.
  • vptr (Virtual Pointer): A hidden pointer inside the object that points to the vtable. Created at runtime when the constructor runs.
  • Virtual Destructor: ALWAYS make base class destructors virtual! Otherwise, deleting a derived object via a base pointer will only call the base destructor, causing a memory leak.
  • override & final: override tells the compiler to check if you are actually overriding a base function. final prevents a function from being overridden further, or a class from being inherited.
  • Object Slicing: Happens when a derived object is passed by value to a base object. The derived-specific data is "sliced" off. Always pass by reference or pointer.

Inheritance

  • Access Specifiers: Public (visible everywhere), Protected (visible in class and derived classes), Private (visible only in class).
  • Constructor Order: Base class constructor runs FIRST, then Derived. Destructor order is exact opposite (Derived, then Base).
  • Diamond Problem: Multiple inheritance where D inherits from B and C, and both B and C inherit from A. D gets two copies of A!
  • Virtual Inheritance: Solves the Diamond Problem by ensuring only one instance of the base class (A) is shared (class B : virtual public A).

Standard Template Library (STL)

Provides ready-to-use data structures and algorithms.

  • vector: Dynamic array. Continuous memory. Fast random access. Slow insertion in the middle.
  • list: Doubly linked list. Fast insertion/deletion anywhere. Slow access.
  • deque: Double-ended queue. Fast insertion/deletion at both front and back.
  • map / set: Implemented as Red-Black Trees. Elements are sorted automatically. O(log n) lookup.
  • unordered_map: Implemented as Hash Tables. O(1) average lookup. Not sorted.
  • queue / stack: FIFO / LIFO container adaptors.
  • priority_queue: Implemented as a Heap. Keeps the largest/smallest element at the top.
  • Algorithms & Iterators: std::sort, std::find. Iterators bridge containers and algorithms securely.

Templates & Exception Handling

Templates

Allows generic programming. Template Specialization: Providing a specific implementation for a particular data type (e.g., making a specialized template just for char*). Variadic Templates: Templates that accept a variable number of arguments.

Exception Handling

try: Code that might fail. throw: Throws the error. catch: Handles the error.

noexcept: A promise that a function will not throw an exception. If it does, the program terminates immediately. Great for move constructors.

Move Semantics (C++11)

Before C++11, passing heavy objects by value resulted in expensive deep copies. Move semantics allow us to "steal" resources from temporary objects.

  • Lvalue vs Rvalue: Lvalue has a memory address (e.g., a variable). Rvalue is a temporary value (e.g., 5, or the return of a + b).
  • std::move: Casts an lvalue to an rvalue reference (&&), telling the compiler: "I don't need this object anymore, you can steal its guts."
  • Move Constructor: Takes an rvalue reference Class(Class&& other). Simply copies the pointers and sets the other pointers to null. Extremely fast!
  • Perfect Forwarding: Using std::forward in templates to preserve the lvalue/rvalue nature of arguments passed to another function.

Modern C++ (C++11/14/17/20)

  • auto: Compiler deduces the type automatically.
  • decltype: Extracts the type of an expression without evaluating it.
  • nullptr: Replaces NULL. It is a type-safe pointer literal, avoiding overload resolution bugs.
  • Range-based for: for (const auto& item : vector). Clean and readable.
  • enum class: Strongly typed enums. Values don't implicitly convert to integers and don't leak into the surrounding scope.
  • Structured Bindings (C++17): auto [x, y] = getCoordinates(); easily unpacks pairs, tuples, or structs.
  • consteval (C++20): Strictly forces a function to run at compile-time (unlike constexpr which can fall back to runtime).
  • Concepts (C++20): Allows constraints on template parameters (e.g., template <Integral T>).

Multithreading (C++11)

  • std::thread: Spawns a new hardware thread. Must call .join() (wait to finish) or .detach() (run independently).
  • std::mutex: Prevents race conditions by allowing only one thread to access shared data at a time.
  • std::lock_guard: RAII wrapper for a mutex. Locks on creation, unlocks on destruction. Exception safe!
  • std::unique_lock: Like lock_guard, but you can manually lock/unlock it, and use it with condition variables.
  • std::condition_variable: Allows a thread to sleep and wait for an event/signal from another thread.
  • std::atomic: For simple types (like int), guarantees lock-free atomic operations.
  • std::future & std::promise: A way to return a result from a background thread asynchronously.

Common Interview Questions

  • Copy constructor vs Assignment operator: Copy constructor is called when a NEW object is created from an existing one. Assignment operator is called when an EXISTING object is replaced by another.
  • Deep vs Shallow copy: Shallow copies only the pointer address (both point to the same memory). Deep copies allocate new memory and copy the actual values.
  • Virtual vs Pure virtual: Virtual has an implementation that CAN be overridden. Pure virtual (= 0) HAS NO implementation and MUST be overridden.
  • Override vs Overload: Overload is same function name, different parameters (compile-time). Override is same function name and parameters in a derived class (run-time).
  • Why does object slicing happen? Because base classes are smaller. Passing a derived object by value to a base class function chops off the extra derived bytes to fit into the base class size.
  • When is the vtable created? The vtable is created by the compiler at compile-time. The vptr is initialized at run-time during object construction.

Rapid Revision Checklist

  • Prefer smart pointers over raw pointers.
  • Base class destructor MUST be virtual.
  • Pass heavy objects by const reference (e.g., const std::string&).
  • Always use the override keyword for virtual functions to catch typos.
  • Prefer std::make_unique and std::make_shared over new.
  • Avoid cyclic shared_ptr references by using weak_ptr.
  • Use RAII for all resources (Memory, File handles, Mutex locks).
